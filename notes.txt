# PortDesk Keyboard Input Solutions Notes

## Quick Fix (Priority 1)
- إذا الكيبورد بشكل عام في PortDesk لا يستجيب، جرب تشغيل السيرفر من ملف التشغيل الخاص (start_portdesk.bat) أو تشغيله بشكل مخفي/minimized.
- في ويندوز: افتح `start_portdesk.bat` (مُعد لـ `start /min python "portdesk-server.py"`). هذه الطريقة تحل مشكلة focus/input في الأغلب.
- في لينكس وماك: استخدم السكربتات `start_portdesk_linux.sh` أو `start_portdesk_mac.sh` (أدناه) لتشغيل السيرفر في الخلفية.

## Overview
This document explains the keyboard input handling solutions implemented in PortDesk to address issues where mobile keyboard input reaches the server but doesn't display on the target system (e.g., Windows, Linux). The solutions provide layered fallbacks for reliability across different environments.

## Problem Description
- Mobile keyboard input is sent to the server successfully.
- Server cursor responds (indicating input reception).
- But characters don't appear on screen intermittently.
- Issue occurs due to focus, window state, input method, or environment differences.

## Windows-Specific Issues and Solutions

### Potential Problems:
- **Focus Issues**: Keyboard input may not register if the target application window is not in focus or is minimized.
- **User Account Control (UAC)**: Windows UAC can block or prompt for input simulation attempts, especially when running as a standard user.
- **Antivirus Interference**: Some antivirus software may detect pyautogui as suspicious and block its operations.
- **Virtual Machine Restrictions**: In virtualized environments, input simulation might be restricted or unreliable.
- **IME (Input Method Editor)**: Issues with non-English input methods or special characters.

### Solutions Implemented:
- **Primary Method**: Uses pyautogui for cross-platform keyboard simulation, which works reliably on Windows.
- **Shortcut Handling**: Custom `_press_win_shortcut()` function maps common shortcuts to Windows equivalents.
- **Fallback Mechanisms**: If pyautogui fails, the system logs errors and suggests manual intervention.

### Requirements:
- Windows GUI environment (not headless).
- pyautogui package installed (already included in requirements).
- Administrator privileges recommended for full functionality.

### Troubleshooting Steps:
1. Ensure the target window is focused and not minimized.
2. Run the PortDesk server as Administrator to bypass UAC restrictions.
3. Temporarily disable antivirus real-time protection for testing.
4. Check Windows Event Viewer for any blocked operations.
5. Test with simple text first, then shortcuts.
6. Verify that the Python environment has access to user input devices.

## macOS-Specific Issues and Solutions

### Potential Problems:
- **Accessibility Permissions**: macOS requires explicit accessibility permissions for applications to simulate input, which PortDesk may not have by default.
- **Focus and Window Management**: Input may fail if the target application is not active or if macOS security features intervene.
- **Gatekeeper Restrictions**: Unsigned or unnotarized applications might be blocked from simulating input.
- **Key Mapping Differences**: Windows-style shortcuts (e.g., Ctrl+C) need to be mapped to macOS equivalents (e.g., Cmd+C).
- **SIP (System Integrity Protection)**: May restrict certain system-level input simulations.

### Solutions Implemented:
- **Primary Method**: Uses pyautogui with platform-specific key mappings.
- **Shortcut Translation**: Custom `_press_mac_shortcut()` function translates Windows shortcuts to macOS Command-based equivalents.
- **Permission Handling**: The system attempts input simulation and logs permission-related errors for user resolution.

### Requirements:
- macOS with GUI (not headless).
- pyautogui package installed.
- Accessibility permissions enabled for the Python/Terminal application running PortDesk.

### Troubleshooting Steps:
1. Enable Accessibility permissions: Go to System Preferences > Security & Privacy > Privacy > Accessibility, and add/check Terminal or Python.
2. Restart the PortDesk server after granting permissions.
3. Ensure the target application is in focus.
4. Test with simple text input first.
5. Check Console.app for permission denied errors.
6. If using a virtual environment, ensure the Python executable has permissions.
7. For Gatekeeper issues, right-click the script and select "Open" to bypass restrictions.

## Linux-Specific Issues and Solutions

### Potential Problems:
- **Root Access for uinput**: Virtual keyboard requires root privileges, which may not be available.
- **Wayland Compatibility**: xdotool doesn't work on Wayland, limiting fallback options.
- **Missing Dependencies**: uinput or xdotool packages may not be installed.
- **Headless Environments**: No GUI means input simulation fails.
- **Environment Variables**: DISPLAY or WAYLAND_DISPLAY not set properly.

### Solutions Implemented:
1. **Virtual Keyboard Device (Primary)**: Uses uinput to create kernel-level input device for direct keystroke injection.
2. **xdotool Fallback (Secondary)**: Command-line tool for X11-based input simulation.
3. **pyautogui Fallback (Tertiary)**: Cross-platform library as final backup.

### Requirements:
- Linux kernel with uinput support.
- `python-uinput` package: `pip install python-uinput`.
- Root privileges for uinput.
- X11 display server for xdotool.
- `xdotool` installed: `sudo apt install xdotool`.
- pyautogui package (already included).

### Troubleshooting Steps:
1. Check console output for error messages.
2. Verify environment variables (DISPLAY, WAYLAND_DISPLAY).
3. Ensure required packages are installed.
4. Try running with root privileges for uinput.
5. Check if running in headless mode.
6. For Wayland: Use XWayland if available, or rely on uinput/pyautogui.
